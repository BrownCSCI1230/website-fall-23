Mentor Meetings Guide

As a CS 1230 TA, you will be meeting with your assigned students on a weekly basis in order to gauge their progress in the course and understanding of the material through project grading and, optionally, lab checkoffs.

Before your meetings

After 72 hours past the deadline (Sunday 10:30 AM) and at least 24 hours before your meetings, for each student:

  1. Verify that the student has submitted on Gradescope. You may search by their email in "Grade Submissions" > hover over "1.1: GitHub repository" > click "Submissions".

    • If they have not submitted, check if the student has an extension in the "Extensions" on the Gradescope assignment for the project.
      • If they do, choose one of the following:
        1. Schedule a one-off meeting after they finish
        2. Cancel this week's meeting and make a plan to fit this week's material into next week.
        3. Ask an HTA to schedule a one-off meeting with the student
      • If they do not, let an HTA know.

  2. Open the student's repository on GitHub in a browser tab at the commit id they submitted on Gradescope.

  3. Do initial grading on the student's submission.

Swaps

Please look ahead to know as early in advance if you must swap for a given week. If so, post in the #swaps channel on Slack to see if there is a TA who can swap with you.

If an urgent situation arises and you cannot make your scheduled meetings at the last minute, let an HTA and your students know as soon as possible. An HTA will help you reschedule your meetings for that week.

Initial grading

Overview

Before your block of mentor meetings and after the late deadline, you will do an initial quick grading pass on each student's submission. In this initial pass, you will give points as conservatively as possible, only for output which is clearly correct. For cases which warrant partial credit, make a note to discuss with the student during your meeting.

Algo weeks

For algo submissions, you may choose not to do an initial pass since grading them is quick, does not require per-problem allocation of points, and can be done fully within the meeting.

If you choose to do an initial pass, simply check the student's final answer and make sure a reasonable amount of work is shown. If the student has a correct answer but little work shown, make sure to ask the student to explain their thought process during the meeting.

Code weeks

For code submissions, this initial pass consists of comparing expected output and student output in the corresponding section of the submission-[project name].md file at the repository root and filling out the rubric accordingly. For the Ray projects (Intersect and Illuminate), you may want to use the grading assisant script to make these comparisons easier.

In addition to comparing outputs to evaluate correctness of functionality, you should briefly evaluate the submission in software design and fill out the corresponding rubric items.

Also be sure to give credit for filling out the submission template.

During your meetings

Algo

Go through the student's submission.

If the student has a correct answer and reasonable amount of work, feel free to move on.

If the student has a correct answer but little work shown, make sure to ask the student to explain their thought process. If the student cannot give a reasonable explanation, move on but refer this to an HTA.

If the student has an incorrect answer, ask them to explain their thought process and guide them towards the correct understanding.

Spot check

Refer to the #hours channel on Slack for the spot check of the week. Ask the student to explain their thought process for the spot check problem if you have not already done so in the previous step.

If a student fails the spot check, refer the incident to an HTA.

Code

Opening the project

In the student Mentor Meeting guide, students are given the following instructions for opening their project. Follow along with them and make sure the commit id they are on is the same as the one they submitted on Gradescope.

  1. Open a terminal and navigate to the project directory.
  2. Run git status to make sure you are on the correct branch and have no uncommitted changes. If you have uncommitted changes, commit them or stash them with git stash. You can unstash them later with git stash pop.
  3. Run git checkout <commit id>. This will put you in a detached head state, which is fine.
  4. Run git status again to make sure you are on the correct commit.
  5. Run qtcreator . to open the project in Qt Creator. If you do not have the qtcreator command installed, you can open the project by opening Qt Creator and selecting File > Open File or Project... and selecting the project directory.

Spot check

Do the spot check at the beginning, especially if the task is time-consuming, such as rendering a scene.

Refer to the #hours channel on Slack for the spot check of the week. Ask the student to run their code according to the spot check and verify that it matches the output in their submission.

If a student fails the spot check, refer the incident to an HTA.
